# Map Hazards — Sprint Plan & Design Doc

Living source of truth for the multi-batch effort to add new **map-authorable hazards** to Chao Chao. Each batch is one PR. Agents pick up the next batch by reading this doc + the handover prompt. **Update the Status Log + check off boxes in the same PR that ships a batch.**

> Sibling reference: the per-file architecture index is `ARCHITECTURE.md` (see the `hazards.js` row for the add-a-kind checklist). Player-facing behavior lives in the Codex (`client/scripts/learn.js`). Tuning lives in `server/config.json`.

---

## 1. Goal

Grow the roster of map-placeable hazards on top of the **hazard-kind registry** that shipped in v0.35.1. Hazards are placed in the in-browser map editor and stored in map JSON as `{ id, x, y, [angle] }`. The registry makes each new kind a localized addition — no edits to the generic spawn/wire/validate paths.

## 2. The framework — how to add a hazard kind

Single source of truth: `server/entities/hazards.js` (`HAZARD_KINDS` / `registerHazardKind` / `hazardKindById`). Adding a kind touches exactly these spots:

1. **`server/config.json` → `hazards.<key>`** — id + tuning. **Next free id: 912** (900–905 = bumper/movingBumper/bumperWall/rotor/geyser/mine; **906 antlion, 907 thumper are taken** by the antlion brutal mode, not registry kinds; **908 vortexWell** = Batch 2's force zone — the Gust Fan was scrapped before shipping, see §5; **909 laserGate, 910 crusher** = Batch 3; **911 sentryTurret** = Batch 4).
2. **`server/entities/hazards.js`** — a `Hazard`/`Rect` subclass + one `registerHazardKind(key, { railed, directional, build })` call.
3. **`client/scripts/draw.js`** — a drawer in `buildHazardDrawers` (id → fn).
4. **`client/scripts/create.js`** — an entry in `EDITOR_HAZARD_KINDS` (palette button, swatch, paint, shortcut). Optional hooks: `paint` / `swatchPaint` / `segmentSelect`.
5. **`.github/scripts/<kind>-test.js`** — a real-engine headless test, wired into `.github/workflows/pr-validation.yml`.
6. **Docs:** `CHANGELOG.md` (`## Unreleased`, player-voice bullet — CI-enforced for config/game/engine changes), `client/scripts/learn.js` Codex card/mention, `ARCHITECTURE.md` hazards row, and check a box here.

Everything else is registry-driven and needs **no** change: `gameBoard.generateHazards`, `utils.validateMap` (`directional` ⇒ finite angle), the Lightning speedup, the editor palette/thumbnail/keyboard plumbing, and the wire payloads.

### Registry contract
- `railed` — rides a `HazardRail` (implies directional). `directional` — map entry needs a finite `angle` (validateMap rejects otherwise; non-finite ⇒ NaN math). `build(entry, mapID, roomSig)` — construct from the JSON entry.
- **Per-tick motion:** moveable kinds either ride a `rail` *or* define `advance(dt)` (called by `engine.updateHazards`, which has the tick dt). Static kinds set `moveable=false`.
- **Per-tick state:** stationary stateful kinds run a phase timer in `update(dt)` (`gameBoard.updateHazards` passes dt) and ship a `netState` int.
- **Lightning:** moving kinds implement `scaleSpeed(mod)` (rail → speed, rotor → angularSpeed); timed/static kinds omit it.

### Wire slots (already built, in `compressor.js` + `client/scripts/gameboard.js`)
- Per-tick row `[ownerId, x, y]` + optional `[3] angle` (only if `hazard.streamAngle`) + optional `[4] netState`. Bumpers stay 3 fields.
- Creation row (`newHazards`) `[ownerId, id, x, y, angle, railX, railY, netState, radius]`; railed kinds ship the **rail's** origin/angle. Slot `[8] radius` is **opt-in** (like `streamAngle`/`netState`): a kind sets `sizable=true` to ship its per-instance authored radius (the vortex well); it's `null` for every other kind.
- Client smooths position (`tx/ty`) and streamed angle (`ta`, via `smoothAngle`, shortest-arc).

### Damage patterns available (reuse, don't reinvent)
- **Map-owned Punch** (radial knockback): set `this.punch = new Punch(x,y,radius,color,ownerId,roomSig,bonus,false,null)`; mark `mapOwned=true`, `type="bumper"` (keeps the pinball-medal tally + team-gate exemption). `gameBoard.updateHazards` emits it; it collides with players for ~100ms. Used by bumper, wall, rotor, geyser, mine.
- **Explosion force** (`gameBoard.applyExplosionForce`) — coupled to projectileList; prefer the Punch path for hazards.
- **Proximity trigger** — give the hazard a collision `radius`; `handleHit(player)` fires when a kart overlaps (used by the mine).

### AI awareness (`server/aiController.js`)
Two layers: (a) `hazardRepulsion` (per-tick steering away from live hazard positions — generic radial field handles most circle hazards) and (b) the path-cost classification loop (~line 1920) sorting hazard cells into `staticHazardCells` (harsh) vs `railCells` (mild/timeable). Add a branch there only if the default point-nearest-cell classification is wrong for the kind's shape (rotor uses `isRotor` → swept-ring cells as railCells; wall uses `isWall` → segment cells as static).

### Testing pattern
Copy an existing `<kind>-test.js` (rotor/geyser/mine are the cleanest templates). Boot the real server headlessly, mock `Date.now`+`setTimeout` into a per-tick clock, seed `Math.random` (mulberry32). Cover: (A) pure geometry/state machine, (B) the wire-payload shape, (C) a live `room.update(dt)` loop proving the actual gameplay outcome (bonk/launch/boom). Wire into `pr-validation.yml`.

## 5. Per-batch design specs

### Batch 2 — Vortex Well  ✅ SHIPPED (the new primitive: a force ZONE, not a punch)
> **Scope change:** this batch was originally Gust Fan **+** Vortex Well. The **Gust Fan was cut** during playtest — the operator is building a boon too similar to it (a directional wind/speed effect). It was fully built then removed cleanly (config/registry/draw/editor/AI/classifier/test); id 908 was reassigned to the Vortex Well. If a wind hazard is ever wanted, the rotated-Rect `forceZone` pattern is in git history on this branch (commits before the cut).

The genuinely new bit vs batch 1: apply a constant per-tick force to players **inside a region** rather than via the Punch system. **How it landed (and a correction):** the first cut reused the Dash Arrows trick — apply the force in `handleHit` (which `engine.narrowBase` calls when a kart overlaps). That was WRONG for an *uncapped* force: narrowBase calls `handleHit` up to **twice** per overlapping pair (once with each object as the outer loop var), and whether both fire depends on quadtree node placement — so the force was ~2× and non-deterministic. The fix is the dedicated path the spec originally suggested: a `forceZone: true` flag + an `applyForce(player)` method, called **once per hazard per tick** over the player list in `gameBoard.updateHazards` (handleHit is a no-op for these). No engine/compressor change; static, so the per-tick wire row stays 3 fields and `newHazards` carries it. The force is a fixed per-tick increment (not dt-scaled), tuned **below the kart's own thrust** so driving always wins; drag bounds it; `maxVelocity` caps it; protected/star-power/finished karts are skipped (`applyExplosionForce` policy); applied only while racing/collapsing.
- **Vortex Well** (id 908, `directional: false`) — a circular **Hazard** (`isVortex`); `applyForce` pulls toward the core with a **calm-eye** profile: `pull = force·4·r·(1−r)` where `r = dist/radius` — **zero at the dead centre and the rim, peak (`force` = 7) in the mid-ring**. This is the load-bearing design choice: the original "peak at centre" linear ramp was a **roach motel** (the strongest pull pointed exactly opposite your only escape, so a stopped kart could never drive out — verified by sim, and caught by the operator in playtest: *"how do you get out of the middle?"*). The calm eye lets you build speed in the quiet centre and punch out through the ring; `force` 7 sits below thrust (~12.5/tick) so driving out always works (worst case ~3.8s flooring straight out from a dead stop). Carry speed → slingshot past; crawl → drawn toward the eye. AI: the generic radial `hazardRepulsion` (h.radius) keeps bots off it; the classifier penalizes the inner-`0.6·radius` core cells (STATIC) so A* routes around the centre.
- **Blur effect:** the interior renders as a hazy violet swirl (`draw.js drawVortexWell`). The blur is baked ONCE into an offscreen sprite via `ctx.filter = "blur()"` (the `getBlackoutHoleSprite` pattern) and rotate-blitted each frame (scaled to the instance radius) — a per-frame canvas filter is a mobile GPU killer (see the cosmetic-perf-collapse history), so the filter never runs in the live loop. It's a stylized haze, NOT a true frosted-glass blur of the terrain/karts behind it (that would need a per-frame framebuffer capture+blur — deferred as too costly).
- **Brutal-round interactions:** a ZOMBIE (infection) is an alive kart in playerList, so the well pulls it like any kart — consistent with `applyExplosionForce` (both skip only protected/star-power); zombies are lava-immune so a lava-core well can't cheese-kill them. An ANTLION (antlion round) is a hazard steered by its own `newX/newY` chase loop (never `velX/velY`), so the kart force-zone can't touch it — instead `updateAntlionRound` adds a dedicated inward pull (same calm-eye profile, capped at `antlionPull`·`chaseSpeed` < chaseSpeed so the chase always wins on the far side → dragged/curved, never trapped; a well over non-sand just leashes it off-habitat and it burrows out). Thumpers stay immovable (fixed sanctuaries). Covered by `antlion-test` (A/B vortex-pull session) + `vortex-well-test` (zombie pulled, non-player ignored by the kart path).
- **Trail suck-in (visual):** `gameboard.js updateVortexTrailPull(dt)` runs per frame and drags every kart's older trail vertices that sit inside a well toward its core (radial pull + tangential swirl, both scaling with depth) so the tail visibly spirals in; the same pass slows those vertices' aging (nudging their `t` toward `now`) so the trail LINGERS in the well. Reuses the existing trail age→alpha→trim machinery untouched (a swirled, persistent vertex is just an old vertex that moved and aged slowly) — no change to `drawTrail`/`trailEffects`. Tunables: `VORTEX_TRAIL_PULL`/`_SWIRL`/`_HOLD`.
- **Per-instance size (drag-to-resize):** the well's `radius` is authored PER INSTANCE — config `radius` (150) is the MAX, `minRadius` (70) the floor, and a fresh placement defaults to the **midpoint** (110). The editor exposes a **resize handle** (the `resizable` kind flag) in place of the rotate knob (the well is radially symmetric — spinning it did nothing): drag it to size the pull radius, clamped to `[minRadius, radius]`. The size rides the map entry (`{id,x,y,radius}`), is **clamped server-side** in `vortexWellRadius()` (a crafted map can't ship a giant well), and ships to clients in the **opt-in `newHazards` slot `[8]`** (gated by the kind's `sizable` flag; per-tick rows stay 3 fields — size is static). `coreRadius` (the drawn centre + the calm eye) scales with the instance radius. The strong-pull core fraction (`coreFraction` 0.6) lives in config, shared by `aiController` and `mapClassifier` so the bot routing and the fairness/par estimate never drift. AI (`hazardRepulsion` h.radius, classifier `radius·0.6`) already reads the live per-instance radius.
- Tests (`.github/scripts/vortex-well-test.js`, wired into `pr-validation.yml`): [A] inward pull + calm-eye-peaks-in-the-ring + peak≤force + rim-cutoff + protected/star/finished + handleHit-noop, [B] static 3-field wire + `newHazards`, [C] live loop — **exactly one pull/tick (not doubled)**, parked kart drawn toward the core, and the **escape regression: a kart flooring out from the dead centre drives clear of the well**.
- **Deferred polish:** custom in-game SFX (none — silent like batch-1 hazards); the editor selects/rotates via the default disc handle; the well is still pushed into the collision quadtree (handleHit no-op) — harmless but slightly wasteful; a true framebuffer blur of content behind the well (vs the stylized haze).

### Batch 3 — Laser Gate + Crusher  ✅ SHIPPED (the new primitives: a timed passability gate + a railed Rect)
The genuinely new bits vs prior batches: (1) a hazard whose **collision turns on and off** on a timer, and (2) a moving **rotated Rect** (every prior moving kind was a circle bumper/rotor; every prior Rect was static).

- **Laser Gate** (id 909, `directional: true`) — a laser barrier strung between two pylons, cycling **OPEN → WARN → SOLID → OPEN** (`openMs`/`warnMs`/`solidMs`) on a phase timer in `update(dt)`, shipping the phase on the `netState` wire (the geyser pattern). A thin rotated `Rect` like the bumper wall (so the kart collides with the **beam**, not a disc — same `getVertices`/`getExtents`/centerline `ax..by` contract). **SOLID = a non-lethal BOUNCE, not a burn** — the deliberate design call: the stated primitive is a *passability gate*, a bounce is recoverable, and a hidden-timing DEATH from being shoved into a lit beam reads unfair (place a fence over lava if you want lethality from the shove). The bounce is `engine.bounceOffSegment` — the **rotated-segment cousin of `bounceOffBoundry`** (`bounceOffBoundry` is AABB-only, so it can't reflect off the arbitrary pylon axis): it reverts the crossing move (no tunnelling), reflects the velocity component normal to the beam (with `restitution`; tangential slip is kept so a kart slides along the barrier), and ejects a kart the beam solidified on top of. OPEN/WARN are passable (`handleHit` no-op); only SOLID collides. AI: `isLaserGate` — the `blocking` flag (warn|solid) holds bots off the beam line only while it's closing (`hazardRepulsion`), and the classifier prices the beam cells at the **mild TIMEABLE `railCells`** penalty (there's a window to cross), not the harsh static-wall price.
- **Crusher** (id 910, `railed: true`) — a heavy slab sliding a corridor on a rail (a Thwomp). It **rides a `HazardRail`** (so the registry/wire/editor treat it as railed — `newHazards` ships the rail origin/angle, the client draws the track) but **owns its motion via `advance(dt)`** rather than the engine's generic rail branch: that branch overloads `hazard.angle` to track rail direction, which would fight the slab's fixed broadside orientation, so the crusher steps the rail param itself (the same overshoot-proof parametric scheme — clamp the scalar `t`, reflect at both ends) and tracks direction in a separate `dir`. The slab is a **CENTERED** rotated `Rect` (broadside to the rail; a custom `getVertices` straddling `(x,y)`, unlike the bumper wall which anchors at one end), collided with the bumper-wall rotated-rect machinery; `move()` re-derives the vertices + centerline each tick. **Contact splits:** mid-rail it's a hard directional **shove** (a map-owned `Punch` off the slab face, like the bumper wall — escapable); the slab slamming home in the outer **`pinchFraction`** of its travel (moving outward) is a lethal **`killSelf("crush")`** (the "pinned against the boundary" pinch — skips protected/star-power karts; both branches idempotent under the up-to-2× `handleHit`). Streams only its rail position (3-field rows — **no streamAngle**, the orientation is fixed and the client derives the broadside from the rail angle). **AI: free** — it's railed (`h.moveable && h.rail`), so `aiController`'s moving-bumper rail-crossing logic (`railCrossingOpen`, end-bias) and the `railCells` mild penalty already time the gap; no new branch.
- **Editor:** both get a `paint`/`swatchPaint` and a config `radius` purely for the editor's select band (the live `Rect` instances never set `this.radius`, so collision stays on the rect path — the bumper-wall lesson). Fence is `segmentSelect` (click the beam); crusher selects at its anchor disc (the rest position) like the moving bumper. `mapClassifier.hazardAvoidance` penalizes each one's whole **lane** (fence beam length / crusher rail length) so the par/fairness estimate routes around them like the moving bumper / bumper wall.
- Tests (`.github/scripts/laser-gate-test.js`, `crusher-test.js`, wired into `pr-validation.yml`): fence — [A] phase machine + `blocking` + the solid-only bounce (revert+reflect) + open/warn pass-through, [B] `netState` wire + creation row (angle at `[4]`, phase at `[7]`), [C] live loop — a kart **blocked by the solid beam vs driving clean through the open one**. Crusher — [A] rail oscillation (clamped, reflects, never off-rail) + centered rotated-rect geometry + shove-vs-pinch-crush + protected/star immunity + `scaleSpeed`, [B] railed creation row (rail origin/angle) + 3-field per-tick rows, [C] live loop — a kart **crushed at the slam point vs shoved-and-survives mid-rail**.
- **Deferred polish:** custom in-game SFX (none — silent like the other batch hazards); the editor previews the fence in its OPEN look (the live blink is in-game only); the static map thumbnail (`gameboard.js`) draws both as generic discs like the other batch-1/2 kinds. A future kind wanting a moving rotated slab can lift the Crusher's centered-`getVertices` + `advance`-driven rail pattern.

### Batch 4 — Sentry Turret (id 911)  ✅ BUILT (the new primitive: a projectile EMITTER)
The genuinely new bit vs every prior batch: this one **fires a real projectile**. Batches 0–3 all hit via the Punch/force machinery on contact; the turret lobs a shot that flies and resolves at range. It's stationary and tracks the nearest racer in its **mount arc**, charging (a dodge-window telegraph) then firing every cooldown. It opts into **both** framework wire slots: `streamAngle` for the live barrel facing (rotor pattern) + `netState` for the idle→charging→firing phase (geyser pattern). Directional — the authored `angle` is the mount facing (the centre of the firing arc).

- **The shot rides the existing projectile wire — no new hazard wire field.** `TurretShot extends Projectile` and ages on the per-tick `projList` wire + `terminateProj` cleanup like any projectile, spawned via one new event (`spawnTurretShot`, carrying x/y **and the fire angle** since a hazard has no owning player to read a position/heading from). `gameBoard.fireTurret` keys it by the turret's mapID (never collides with a player's projectile). **It has its OWN identity, not the iceCannon's:** a unique `type: "turretShot"` drives a bespoke red **energy-bolt** drawer (`draw.js`, oriented along the shipped fire angle), a synthesized **fire** SFX (`audio.js playTurretFire` — an electric pew, distinct from the bomb/ice shot) on `spawnTurretShot`, and a unique red **impact** burst + `playTurretImpact` thunk on the dedicated `turretShotBurst` detonation event (not the iceCannon's `snowFlakeExploded` ice flash). The distinct type also means the engine's `snowFlake`-only cell-collision is skipped for it (it flies through terrain — wanted).
- **SHOVE, not freeze (the decision the handover asked for).** Reusing iceCannon's *freeze* was the stated default, but the iceCannon snowflake ices **every cell it crosses each tick** (`handleHit(cell)` → tileChanges) **plus a 100px burst** on detonation — fine for a one-shot ability, but an auto-firing turret would runaway-ice the whole arena and unpredictably reshape any map it's on. So the turret shot **mutates no terrain** and resolves as a recoverable knockback: it detonates on the first live racer it touches (a tracked shot connecting) OR on its fuse (the max-range failsafe), sets a `shove` flag, and `gameBoard.shoveShot` applies the existing `applyExplosionForce` knockback (no `explodeIce`, no tileChanges). Recoverable + bounded + consistent with every other hazard's force-based contact; a kart shoved into lava/off a ledge is the emergent danger (the map author's choice of placement), unattributed (no playerList entry for the turret owner). If the operator ever wants the freeze instead, it's a localized flip (set `explodeIce` on the shot + key it so `explodeIce(owner)` runs).
- **Aim + state machine need the player list**, which the stationary `update(dt)` hook doesn't get — so they live in `Turret.serve(playerList, dt, live)`, called from `gameBoard.updateHazards` (where the geyser's punch + the vortex's force already run). `serve` acquires the nearest in-arc/in-range racer (skipping protected/star/finished), eases the barrel toward it (clamped to `turnSpeed`), walks idle→charging→firing on the `cooldownMs`/`chargeMs`/`fireMs` timers, and raises `fireRequest` on the firing edge (mirroring the abilityList `spawnSnowFlake` flag → `gameBoard.spawnSnowFlake` pattern). **Baiting works:** a target that jukes out of the arc DURING the charge aborts the shot — the telegraph is a real dodge window (the laser-gate warn / geyser charge philosophy). Idle (no charge) while not racing. Starts "loaded" (cooldown pre-elapsed) so a turret guarding the start threatens promptly. No `scaleSpeed`, so Lightning leaves it alone (timed/stationary, like the geyser).
- **Also targets antlions (operator follow-up).** Antlions (the chasing brutal-round hazard) are valid turret targets — `gameBoard.updateHazards` collects them (gated on the antlion round) and passes them to `serve`/`acquireTarget`, which picks the nearest of {racers, antlions} in arc/range/LoS (shared `targetAngleIfVisible` helper). The bolt detonates on an antlion (`TurretShot.handleHit` → `isAntlion`); since antlions chase via `newX/newY` (not `velX/velY`, so the kart explosion force can't move them), the knockback reuses their dedicated **impulse channel** (`impVX/impVY`, the same one the thumper slam uses) via `gameBoard.shoveAntlions`. So a turret near a sanctuary helps fend off the swarm. Zombies (players) were already targeted; antlions (hazards) now are too.
- **Barriers shield you (operator follow-up).** A wall/fence (the Fences & Walls feature's `map.barriers`) blocks the turret: `acquireTarget` skips any racer whose straight line from the turret is broken by a barrier (`engine.barrierCrossing` — the new segment-vs-barrier test), so a walled kart is never even targeted; and an already-airborne bolt is absorbed AT the barrier in `gameBoard.updateProjectiles` (a fizzle — impact FX + sound, NO knockback), catching a kart that ducks behind a wall after the shot is away. So a barrier fully protects whoever's behind it. (Test [D] in `sentry-turret-test.js`: walled kart never fired on + no-wall control still fired on.)
- **AI — a new cone-of-fire shape** (the handover's ask). `aiController` gets an `isTurret` branch in the classifier (~line 2137): it prices the cells inside the firing **wedge** (mount arc, out to `range`) at the **mild TIMEABLE `railCells`** penalty — the turret fires periodically and the shot only SHOVES (recoverable), so a bot can risk crossing between shots rather than wall off the whole wedge (the laser-gate analogy) — using the **fixed mount angle**, not the live barrel. A second `isTurret` term in `hazardRepulsion` pushes a bot **laterally** toward the nearer cone edge when it's inside the wedge (perpendicular to the turret→bot line) so it clears the line of fire while still driving forward rather than backing off. `mapClassifier.hazardAvoidance` samples the cone (centre + both edges, out to range) for the par/fairness estimate, in lockstep with the live AI. Both AI branches are gated behind `isTurret`, so they're **inert on turret-free maps** — ai-fitness A/B is **identical on all three control maps** (crossroads/FastAndSlow/IcyLake).
- **Editor:** an `EDITOR_HAZARD_KINDS` entry (shortcut `s`, directional) + a `paint` that draws the body + barrel at the authored mount angle **with the firing-arc wedge** (the threat cone, out to range) so authors see exactly what the turret covers — and rotate/move come free from the shipped universal handle (#314). Live, the rotating barrel + charge glow are the dodge telegraph (the client doesn't have the mount angle, only the streamed barrel, so the full cone is editor-only — like the laser gate's AI region being server-only).
- Tests (`.github/scripts/sentry-turret-test.js`, wired into `pr-validation.yml`): [A] acquireTarget honors range/arc/protected/finished + barrel easing + the full loaded→charge→fire cycle + the **bait abort** (juke out of the arc mid-charge → no shot) + idle-while-not-live + `muzzle()` geometry; [B] the streamAngle+netState 5-field row (`[3]` live barrel, `[4]` phase) + creation row (mount angle at `[4]`, phase at `[7]`); [C] live loop — the turret spawns from a map entry, **fires on a kart in its arc** (the shot rides `projectileList` as a `snowFlake`, detonates, and **shoves** the kart off its spot) while **leaving the terrain unfrozen** (no `tileChanges`, no iced cells — the shove-not-freeze regression), and **never fires on a kart behind it** (outside the arc).
